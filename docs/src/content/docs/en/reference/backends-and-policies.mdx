A `WorkloadSpec` describes *what* to run. Three fields shape how the platform runs it:

- **`backend`** â€” which serving engine launches the model.
- **`task_type`** â€” the modality the model serves (text, embeddings, audio, â€¦).
- **`execution_policy`** â€” how the workload is scheduled and scaled.

Placement onto hardware is the platform's job â€” a `WorkloadSpec` has no `provider` and no `min_vram_gb`. You declare the workload; the platform reconciles it.

## Backends

The `backend` field selects the serving engine. In code you pass the `Backend` enum; on the wire it is a string.

| Backend enum | Wire string | When to use |
| --- | --- | --- |
| `Backend.OLLAMA` / `Backend.Ollama` | `ollama` | Quick local-style serving and broad GGUF model coverage. Simplest to stand up. |
| `Backend.VLLM` / `Backend.Vllm` | `vllm` | High-throughput production text generation. You control the launch command. |
| `Backend.VLLM_OMNI` / `Backend.VllmOmni` | `vllm-omni` | vLLM for multimodal / omni models. Same command-driven config as `vllm`. |
| `Backend.SGLANG` / `Backend.Sglang` | `sglang` | Structured / programmatic generation workloads on the SGLang runtime. |
| `Backend.LLAMACPP` / `Backend.Llamacpp` | `llamacpp` | Prebuilt `llama-server` for **GGUF** models (OpenAI-compatible). Strong on AMD/ROCm and Apple Silicon; `text2text` only. |
| *(custom backend slug)* | `<your-slug>` | **Your own PyTorch model**, defined in Python and published to your tenant. Use when the model or its I/O doesn't fit a built-in. See [Custom backends](/en/guides/custom-backends/). |

<Aside type="note">
Python uses the `UPPER_SNAKE` members (`Backend.VLLM`); TypeScript uses the `PascalCase` members (`Backend.Vllm`). They map to the same wire strings shown above.
</Aside>

<Aside type="tip" title="Custom backends">
Beyond the built-ins, you can publish a **custom backend** â€” your own model as a small Python class (`setup()` loads it once, `process(job)` runs each request) â€” and target it from a `WorkloadSpec` by its slug (`backend="<your-slug>"`). Its I/O is a free JSON dict, not OpenAI-compatible, so any modality fits. It's private to your tenant. See the [Custom backends](/en/guides/custom-backends/) guide.
</Aside>

### Backend config

The `vllm` and `vllm-omni` backends are command-driven: you provide the `command` that launches the server, and optionally pin `vllm_version`. The backend-specific config on the wire is `{ command, vllm_version? }`.

<Tabs syncKey="lang">
  <TabItem label="Python">
```python title="vllm_workload.py"
from inferencekey import ManagementClient, WorkloadSpec, Backend

mgmt = ManagementClient.from_env(project="acme")

ref = mgmt.ensure(WorkloadSpec(
    name="support-bot",
    slug="support-bot",
    model="meta-llama/Llama-3.1-8B-Instruct",
    backend=Backend.VLLM,
    command="vllm serve meta-llama/Llama-3.1-8B-Instruct --max-model-len 8192",
    vllm_version="0.6.3",  # optional: pin the engine version
))
```
  </TabItem>
  <TabItem label="TypeScript">
```ts title="vllm_workload.ts"
import { ManagementClient, Backend } from "@inferencekey/sdk";

const mgmt = ManagementClient.fromEnv({ project: "acme" });

const ref = await mgmt.ensure({
  name: "support-bot",
  slug: "support-bot",
  model: "meta-llama/Llama-3.1-8B-Instruct",
  backend: Backend.Vllm,
  command: "vllm serve meta-llama/Llama-3.1-8B-Instruct --max-model-len 8192",
  vllmVersion: "0.6.3", // optional: pin the engine version
});
```
  </TabItem>
</Tabs>

<Aside type="tip">
`vllm_version` only applies to `vllm` / `vllm-omni`. The `command` is run verbatim by `vllm`, `vllm-omni`, and `llamacpp` â€” for `llamacpp` it must launch `llama-server` against a **GGUF** model (e.g. `llama-server -hf <repo> --hf-file <file.gguf> -ngl 99 -c 4096`). For `ollama` and `sglang`, leave `command` unset and let the backend launch the `model` you declared.
</Aside>

## Task types (modalities)

`task_type` declares the modality the workload serves. There are 12 task types; the default is `text2text`. The data-plane surface depends on the modality.

| Task type | Modality | Data plane |
| --- | --- | --- |
| `text2text` | Text in â†’ text out (chat / completions). **Default.** | OpenAI-compatible chat/completions |
| `embedding` | Text in â†’ vector out | OpenAI-compatible embeddings (`embed`) |
| `text2image` | Text in â†’ image out | OpenAI-compatible |
| `text2audio` | Text in â†’ audio out | OpenAI-compatible |
| `audio2text` | Audio in â†’ text out (transcription) | OpenAI-compatible |
| `reranker` | Query + documents â†’ ranked order | Async-only (no sync OpenAI route) |
| `classification` | Input â†’ label / scores | Async-only (no sync OpenAI route) |
| `reward` | Input â†’ scalar reward score | Async-only (no sync OpenAI route) |
| â€¦ | Additional modalities (12 total) | â€” |

<Aside type="caution">
`reranker`, `classification`, and `reward` are **async-only** â€” they have no synchronous OpenAI-compatible route. The synchronous `generate_text` / `generateText` and `embed` calls target `text2text` and `embedding` workloads.
</Aside>

### Declaring a task type

<Tabs syncKey="lang">
  <TabItem label="Python">

ref = mgmt.ensure(WorkloadSpec(

    model="BAAI/bge-large-en-v1.5",
    backend=Backend.VLLM,
    command="vllm serve BAAI/bge-large-en-v1.5",
    task_type="embedding",
))

emb = data.endpoint("billing", api_key="ik_live_...").embed(input=["a", "b"])
# emb.embeddings
```
  </TabItem>

## Execution policies

`execution_policy` controls how the workload is scheduled and scaled. There are three policies.

| Execution policy | Behavior | When to use |
| --- | --- | --- |
| `fixed` | Runs with a fixed allocation that does not scale. | Steady, predictable load; reserved capacity. |
| `scheduled` | Runs on a schedule. | Batch / periodic jobs; off-hours work. |
| `autoscaling` | Scales with demand. | Spiky or unpredictable traffic. |

<Aside type="note">
Policy-specific tuning is carried in `execution_policy_config`. Set `execution_policy` to one of `fixed`, `scheduled`, or `autoscaling`; pair it with `execution_policy_config` for the parameters that policy expects.
</Aside>

<Tabs syncKey="lang">

    command="vllm serve meta-llama/Llama-3.1-8B-Instruct --max-model-len 8192",
    execution_policy="autoscaling",

```

### `execution_policy_config` shapes

The policy *name* is the stable contract; `execution_policy_config` carries the per-policy parameters and is validated by the Manager. The keys below are the ones the Manager actually validates.

**`fixed`** â€” a constant allocation:

```json
{ "replicas": 1 }
```

**`scheduled`** â€” a window or cron expression:

```json
{ "cron": "0 2 * * *" }              // or: { "window": "Mon-Fri 08:00-20:00 Europe/Madrid" }
```

**`autoscaling` (cloud)** â€” scales cloud workers between a floor and a ceiling. Worker bounds are `min_workers` / `max_workers` (**not** `min_replicas`/`max_replicas`). A cost ceiling `max_hourly_cost_usd` (> 0) is **required**, and `workers.cloud_pool` needs a non-empty `gpu_pool` of catalog display names:

```json
{
  "min_workers": 0,
  "max_workers": 5,
  "max_hourly_cost_usd": 5.0,
  "workers": {
    "source": "cloud",
    "cloud_pool": {
      "gpu_pool": ["NVIDIA A100-SXM4-80GB"],
      "max_hourly_cost_usd": 5.0,
      "max_instances": 5,
      "allowed_cuda_versions": ["13.0"]
    }
  }
}
```

`gpu_pool` entries are GPU display names from the cloud catalog (dashboard â†’ **GPU Resources**). `allowed_cuda_versions` is an optional CUDA pin. A flat `gpu_pool` at the root is a legacy form that **cannot** carry `allowed_cuda_versions` â€” prefer the `workers.cloud_pool` shape above.

**`autoscaling` (private worker)** â€” scales across your own registered workers instead of cloud GPUs:

```json
{
  "min_workers": 1,
  "max_workers": 4,
  "workers": {
    "source": "private",
    "private_worker_ids": ["wrk_7Yc2..."]
  }
}
```

<Aside type="note">
The config schema may gain fields over time, but these keys are what the current Manager validates. The [workloads guide](/en/guides/workloads-by-policy-worker-modality/) shows them in context per modality.
</Aside>
